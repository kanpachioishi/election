# LINE Official Account Plan

最終更新: 2026-04-16 JST

## 1. 背景

この構想の出発点は、

- 選挙が近いことを急に知った
- 十分な情報を集める時間がなかった
- もっと早く気づけていれば判断材料を持てた

という悔しさにある。

このプロジェクトは、公式情報への導線を整理することに加えて、
「知らないうちに大事な選挙が近づいていた」を減らすことを目指す。

そのための将来案として、公式 LINE アカウントを持つ。

## 2. 目的

LINE で実現したいことは、ニュース配信ではなく選挙の気づき支援である。

目標:

- 住んでいる地域や関心地域の選挙が近づいたことを早めに知らせる
- 候補者一覧、選挙公報、期日前投票、投票所案内などの公式情報にすぐ届ける
- 「気づくのが遅かった」を減らす
- 公式情報を見るまでの手間を減らす

## 3. 非目的

少なくとも初期段階では、次のことはやらない。

- 政治的な立場を推す配信
- 候補者評価や投票先の推奨
- 雑多な政治ニュースの配信
- 長文記事中心のメディア運用

まずは、選挙の存在と公式導線を届けることに集中する。

## 4. 想定する通知

優先度順に並べると、まず欲しいのは次の通知。

### 4.1 選挙が近い通知

- 例: `あなたの地域の市長選まであと30日`
- 目的: 早めの気づき
- 必要データ:
  - 対象地域
  - 投票日
  - 選挙種別

### 4.2 告示通知

- 例: `本日告示。候補者情報と選挙公報の確認先はこちら`
- 目的: 情報収集の開始点を作る
- 必要データ:
  - 告示日
  - 候補者一覧の公式リンク
  - 選挙公報リンク

### 4.3 期日前投票開始通知

- 例: `期日前投票が始まりました`
- 目的: 投票機会を逃しにくくする
- 必要データ:
  - 期日前投票ページ
  - 期日前期間
  - 投票所案内

### 4.4 投票日前日・当日通知

- 例: `明日は投票日`
- 目的: 最後のリマインド
- 必要データ:
  - 投票日
  - 投票所案内
  - 投票時間

### 4.5 公式資料公開通知

- 例: `選挙公報が公開されました`
- 目的: 見るべき資料が出たことを伝える
- 必要データ:
  - 選挙公報 URL
  - 候補者一覧 URL
  - 投票所 URL

## 5. 初期の配信設計

最初から細かい個別最適を目指しすぎない。

初期案:

- 配信対象は `都道府県知事選` と `主要な市長選` から始める
- 通知は `あと30日 / 告示日 / 期日前開始 / 前日` の4本を基本にする
- まずは手動または半手動で送れる設計にする
- 住所の厳密登録ではなく、`都道府県` または `関心地域` の選択から始める

理由:

- 今の dataset でも比較的作りやすい
- 精度確認をしながら運用を学べる
- 送るべき通知の価値が見えやすい

## 6. LINE に必要なデータ

今のプロジェクト資産と照らすと、使えるものと足りないものがある。

すでに使いやすいもの:

- `data/v1/elections.json`
  - 選挙名
  - 種別
  - 告示日
  - 投票日
  - 地域
- `data/v1/election_resource_links`
  - 候補者一覧
  - 選挙公報
  - 期日前投票
  - 投票所案内

まだ整備が必要なもの:

- どの通知をいつ送るかのルール定義
- 地域と配信対象の紐づけ
- 通知に使う短いテンプレ文
- 公式情報公開の検知方法
- `この選挙は LINE 通知対象にするか` のフラグ

## 7. 実装前に決めること

実装前に最低限決めたい論点は次の通り。

### 7.1 誰に送るか

- 全国一律配信にするか
- 都道府県単位で受け取り地域を選べるようにするか
- 将来的に市区町村単位まで絞るか

初期は `都道府県` 選択が最も現実的。

### 7.2 何を対象にするか

- 知事選のみ
- 知事選 + 一部の市長選
- 全国の首長選

初期は `知事選 + 優先度の高い市長選` が無理がない。

### 7.3 どこまで自動化するか

- 完全手動
- 通知候補だけ自動生成
- 自動送信まで行う

初期は `通知候補の自動生成 + 最終確認は手動` が安全。

## 8. 段階的な進め方

### Phase 1: 構想固定

- 送りたい通知の種類を固定する
- 対象選挙の範囲を固定する
- 地域選択の単位を決める

### Phase 2: データ準備

- `LINE通知対象` の考え方を決める
- 配信に必要な URL の優先順位を決める
- 不足リンクの棚卸しをする

### Phase 3: 運用試作

- 手動で通知文を作る
- 小さく試す
- 何の通知が役立つかを確認する

### Phase 4: 実装

- 配信対象管理
- 通知候補生成
- LINE 配信導線

## 9. 今の時点での結論

この構想は、今の dataset 拡張と相性が良い。

今やっていること:

- 選挙の existence を持つ
- 告示日と投票日を持つ
- 公式リンクを持つ

これはそのまま、将来の LINE 通知の土台になる。

したがって、今は LINE 実装を急がず、

- `elections.json` の精度を上げる
- `election_resource_links` を厚くする
- 通知に向く選挙を見分ける

ことを続ければよい。

## 10. 次に着手するときの最小タスク

次回この構想を実装寄りに進めるときは、まず次の 3 つから始める。

1. `LINE で送る通知種類` を 4 本程度に固定する
2. `通知対象にする選挙の条件` を決める
3. `通知に必要なデータ項目一覧` を docs 化する

この 3 つが決まれば、実装の議論に入りやすい。
