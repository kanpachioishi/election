# わたしの選挙 PRD

最終更新: 2026-04-08
ステータス: Draft v2 / MVP境界を固定

## 1. 文書の目的

この文書は、「わたしの選挙」のプロダクト要件を定義するための PRD である。

この PRD で決めること:

- 誰のどんな課題を解くか
- MVP でどこまで作るか
- 何を成功とみなすか
- 実装前に固定すべき要件は何か

この PRD で詳しく決めないこと:

- データテーブルの厳密なスキーマ
- 各画面の詳細 UI デザイン
- 実装技術の細かい選定

## 2. 背景

日本の選挙情報は、自治体の選挙管理委員会サイト、PDF、報道機関の特設ページなどに分散している。

その結果、有権者は次のような問題に直面している。

- 自分の地域で次に何の選挙があるか分かりにくい
- 市長選や知事選は知っていても、議会選挙や補欠選挙を見落としやすい
- 候補者、公報、期日前投票、投票所などの情報が別々の場所に散らばっている
- 公式情報はあるが、見つけにくく、初心者には理解しづらい

実際に、市長選に投票しに行った際に議会の補欠選挙があることをその場で知った体験が、このプロジェクトの出発点である。

## 3. プロダクト概要

### 3.1 プロダクト名

わたしの選挙

### 3.2 一言でいうと

自分の地域の次の選挙がすぐ分かり、投票に必要な公式情報まで迷わずたどり着ける、生活者向け選挙ナビ。

### 3.3 提供価値

ユーザーはこのサービスを使うことで、最短で次のことを把握できる。

1. 自分の地域で次に行われる選挙
2. 投票日と、投票までに確認すべき情報
3. 候補者、公報、期日前投票、投票所などの公式情報への導線

### 3.4 プロダクト仮説

投票率の低さには、関心の問題だけでなく、情報導線の悪さが大きく影響している。

そのため、

- 自分に関係する選挙がすぐ分かること
- 公式情報への到達が簡単であること
- 地方選挙や補欠選挙を見落としにくいこと

を改善できれば、投票行動のハードルを下げられると考える。

### 3.5 MVP の中核約束

MVP では、次の約束だけを確実に守る。

- 郵便番号から、自分の地域に関係する次の選挙へ到達できる
- 各選挙について、最低限の基本情報と一次情報への導線がある
- 出典と確認日時が明示されている
- 確認済みでない詳細情報を、確認済みであるかのように見せない

逆に、MVP では「候補者比較まで全国で完璧に揃える」ことは約束しない。

### 3.6 プロダクトカテゴリの定義

本サービスは、選挙ニュースサイトでも、政治評論サイトでも、投票マッチングサイトでもない。

本サービスのカテゴリは、常設型の生活者向け選挙ナビである。

本サービスの価値は、情報量の多さではなく、自分に関係する選挙へ最短で到達できることにある。

## 4. 解く課題

### 4.1 ユーザー課題

- 「今度、自分の地域で何の選挙があるのか分からない」
- 「選挙があるのは知っているが、候補者や公報をどこで見ればいいか分からない」
- 「期日前投票の場所や期間が探しづらい」
- 「地方選挙や補欠選挙は見落としやすい」

### 4.2 既存サービスのギャップ

- 大手選挙サイトは情報量が多い一方、初心者には複雑
- 報道機関の特設は選挙期間中に強いが、常設で「自分の次の選挙」を出す体験が弱い
- 自治体サイトは一次情報として正確だが、統一感がなく分かりにくい

## 5. ターゲットユーザー

### 5.1 第一ターゲット

- 20代から40代の一般有権者
- 政治ニュースを日常的には追っていない
- 投票する意思はあるが、調べるのが面倒
- スマホ中心で情報収集する

### 5.2 第二ターゲット

- 初めて投票する 18歳、19歳
- 引っ越し後で地域の選挙事情が分からない人
- 家族や友人に共有しやすい選挙情報を探している人

### 5.3 コアユースケース

- 「自分の地域で次の選挙はいつか知りたい」
- 「投票日直前に投票所や期日前投票を確認したい」
- 「候補者や選挙公報を比較したい」

## 6. プロダクト目標

### 6.1 事業・社会的目標

- 「自分の地域の次の選挙が分からない」状態を減らす
- 地方選挙や補欠選挙の見落としを減らす
- 選挙情報へのアクセス障壁を下げる

### 6.2 MVP 目標

MVP の目的は、まず「自分の地域の次の選挙が分かる」を成立させること。

MVP で成功とみなす状態:

- ユーザーが郵便番号起点で、自分に関係する次の選挙へ到達できる
- 各選挙について、最低限の基本情報と公式情報への導線がある
- 出典と確認日時が表示され、信頼できる

MVP の判断基準は、「便利そうか」ではなく、以下を満たすかで判定する。

1. 自分ごとの入口があるか
2. 迷わず一次情報へ進めるか
3. 全国MVPとして運用可能か
4. データ更新を継続できるか

## 7. 非目標

初期段階では、以下は主目的にしない。

- 政治ニュース速報の量で勝つこと
- 候補者への評価やランキングを行うこと
- 選挙マッチングを最初から作ること
- 全国すべての候補者データを完全自動で揃えること
- SNS 的なコミュニティ機能を作ること
- 住所の自由入力を最初から完成させること
- 全選挙で投票所・期日前投票・公報を初期リリースから完全網羅すること

## 8. 成功指標

### 8.1 北極星指標

- 自分の次の選挙に到達できたユーザー率

### 8.2 主要KPI

- 郵便番号検索成功率
- 地域ページ到達率
- 地域ページから個別選挙ページへの遷移率
- 個別選挙ページから公式情報クリック率
- 出典表示率
- 更新日時表示率
- モバイル閲覧完了率
- 地域マッピング未解決率
- 公開ページの未確認率
- 更新SLA超過件数
- 必須フィールド欠損率
- 誤案内インシデント件数

### 8.3 MVP 判定用の定性基準

- 初見ユーザーがトップページで「何のサイトか」をすぐ理解できる
- 郵便番号入力後に、自分向けの選挙一覧が短時間で見える
- 選挙ページで「投票日」と「公式情報への導線」が迷わず見つかる
- データ未整備の項目は、空欄ではなく「未掲載 / 未確認」と分かる

## 9. ユーザーストーリー

### 9.1 生活者として

私は、自分の地域で次に何の選挙があるかを知りたい。
そうすることで、見落とさずに投票準備ができる。

### 9.2 初心者として

私は、候補者や公報、期日前投票の情報があるなら、ひとまとめで見たい。
そうすることで、どこを見ればよいか迷わずに済む。

### 9.3 投票直前のユーザーとして

私は、投票日、投票所、期日前投票の情報をすぐ確認したい。
そうすることで、行動に移しやすくなる。

## 10. MVP スコープ

### 10.1 MVP に含めるもの

- 郵便番号起点の地域特定
- 地域ごとの「次の選挙」表示
- 個別選挙ページ
- 全国選挙スケジュール一覧
- 投票率・過去データの基本可視化
- 出典 URL、確認日時、確認状態の表示
- 公式情報へのリンク導線

### 10.1.1 MVP の具体的な到達ライン

MVP では、以下をプロダクトの完成ラインとする。

- 全国の郵便番号データをもとに、少なくとも都道府県・市区町村レベルの地域特定ができる
- 地域ごとに、関係する次の選挙を一覧表示できる
- 各選挙ページに、基本情報と出典を載せられる
- 候補者、公報、投票所、期日前投票は、確認済みのものだけを個別選挙ページ内でリンクまたは表示できる
- 情報がない場合は、未確認または未掲載として明示できる

### 10.1.2 MVP の公開時カバー範囲

MVP 公開時に保証するカバー範囲は以下とする。

- 郵便番号検索は全国を対象にする
- 国政選挙は upcoming な対象を全国カバーする
- 都道府県単位の選挙は upcoming な対象を全国カバーする
- 市区町村選挙と補欠選挙は、確認・収集できたものから掲載する
- 市区町村選挙と補欠選挙は、未カバーである可能性をプロダクト上で明示する

MVP の目的は、全国の全地方選挙完全網羅ではなく、「全国で使える入口を成立させ、確認済み情報から広げられる運用基盤を作ること」である。

### 10.2 MVP に含めないもの

- 住所の自由入力
- 全国完全網羅の候補者比較表
- 全国完全網羅の候補者詳細ページ
- 全国完全網羅の選挙公報専用ページ
- 全国完全網羅の投票所案内専用ページ
- 全国完全網羅の期日前投票専用ページ
- 投票マッチング
- 通知機能
- アカウント機能
- コメント機能
- 高度なパーソナライズ

### 10.3 MVP の優先順位

#### P0

- 郵便番号検索
- 地域特定
- 地域ページ
- 個別選挙ページ
- 全国スケジュール一覧
- 出典 / 確認日時 / 確認状態

#### P1

- 投票率可視化
- 確認済みの公報リンク
- 確認済みの候補者一覧リンク
- 確認済みの投票所 / 期日前投票リンク

#### P2

- 候補者比較
- 通知
- お気に入り地域
- カレンダー追加

## 11. 必須機能要件

### 11.1 検索・特定

- ユーザーは郵便番号から地域を特定できること
- MVP 時点では住所の自由入力を必須としないこと
- 将来的に住所入力へ拡張しやすい設計であること
- 入力失敗時に、分かりやすいエラーメッセージを表示すること
- 全国の郵便番号データを取り込める構造であること
- 郵便番号が一意に自治体へ解決できない場合は、自動で1つに決め打ちしないこと
- 郵便番号が複数候補にまたがる場合は、候補地域を提示するか、未解決として扱うこと
- 自治体まで確定できない場合は、自治体固有の選挙を断定表示しないこと
- 都道府県まで安全に特定できる場合のみ、都道府県以上で成立する情報を表示してよい

### 11.2 地域ページ

- 地域に関係する次の選挙を時系列で表示すること
- 少なくとも選挙名、種別、投票日、ステータス、出典を表示すること
- 個別選挙ページへ遷移できること
- データが存在しない場合は、その理由を明示すること
- 全国共通のレイアウトで表示できること

### 11.3 個別選挙ページ

- 選挙名
- 選挙種別
- 対象地域
- 投票日
- 告示日または公示日
- 基本説明
- 公式情報リンク
- 出典
- 最終確認日時

を表示できること。

可能な場合のみ、以下を追加表示できること。

- 候補者一覧
- 選挙公報リンク
- 投票所情報リンク
- 期日前投票情報リンク

情報がない場合は、未掲載 / 未確認として表示できること。

### 11.4 スケジュール一覧

- 全国の選挙を一覧で見られること
- 国政、都道府県、市区町村などで絞り込めること
- 地域ページや個別選挙ページへ遷移できること
- 過去日付の表示ルールを持つこと

### 11.5 データ・出典表示

- すべての選挙情報に出典 URL を紐づけられること
- 最終確認日時を保持し、表示できること
- 情報の確度や確認状態を保持できること
- `未確認`, `確認済み`, `要更新` などの状態を扱えること
- `source_url`, `confirmed_at`, `status` は公開データの必須項目として扱うこと

### 11.6 投票率データ

- 過去の投票率推移を閲覧できること
- 年代別などの基礎データを可視化できること
- 出典を明示すること

### 11.7 SEO / 配信前提

- 地域ページと個別選挙ページは URL を持つこと
- 検索エンジンが解釈しやすい HTML を返せる構成に寄せること
- JavaScript 実行後にしか主要情報が出ない設計を避けること

### 11.8 情報鮮度と失効ルール

- 選挙まで 90 日超の情報は、少なくとも 30 日ごとに再確認する
- 選挙まで 90 日以内 31 日以上の情報は、少なくとも 14 日ごとに再確認する
- 選挙まで 30 日以内 8 日以上の情報は、少なくとも 7 日ごとに再確認する
- 選挙まで 7 日以内の情報、または投票所・期日前投票・候補者のような運用情報は、少なくとも 24 時間ごとに再確認する
- 公式情報の更新を検知した場合は、確認期限内であっても `要更新` に落とす
- `confirmed_at` が上記SLAを超過した情報は、自動的に `要更新` 扱いにする
- 選挙終了後の情報は、現役情報として扱わず、アーカイブまたは終了状態へ移す

## 12. リリース判断ルール

### 12.1 リリースを止める条件

以下のいずれかを満たす場合、MVP は未完成とみなす。

- 郵便番号から地域特定できない
- 地域ページから次の選挙へ到達できない
- 選挙ページに出典または確認日時がない
- 古いデータを最新のように見せてしまう
- 一次情報への導線が存在しない

### 12.2 リリースしてよい条件

以下を満たす場合、MVP として公開可能とみなす。

- P0 が揃っている
- P1 は未整備でも、未掲載であることを明示できる
- データ更新方針が定義されている
- 100% の公開選挙ページで `source_url`, `confirmed_at`, `status` が表示される
- 100% の郵便番号検索結果が、`一意解決` または `未解決であることの明示` のどちらかに到達する
- 0 件の公開ページで、期限切れ情報を `確認済み` として表示しない
- 公開カバー範囲と未カバー範囲がユーザーに分かる形で記載されている

## 13. 今後追加する拡張機能

以下は MVP 後の優先候補とする。

- 候補者一覧の標準化
- 候補者比較表
- 選挙公報専用ページ
- 期日前投票専用ページ
- 投票所案内専用ページ
- 通知機能
- カレンダー追加
- お気に入り地域保存

## 14. 非機能要件

### 14.1 信頼性

- 一次情報へのリンクを常に保持する
- 出典のない事実情報は公開しない
- 推測値や未確認情報は明確に区別する

### 14.2 可用性

- 静的配信を基本とし、表示速度を高く保つ
- モバイル回線でも快適に閲覧できること

### 14.3 可読性

- 政治に詳しくない人でも理解できる表現を使う
- スマホで1画面目に重要情報が収まることを重視する

### 14.4 SEO

- 地域ページ、個別選挙ページを検索流入の主軸にする
- title、description、canonical、構造化データを将来的に載せやすい構成にする
- JavaScript だけに依存しないページ生成に寄せる

## 15. コンテンツ要件

### 15.1 基本方針

コンテンツは、意見や論評ではなく「判断材料への導線」を提供する。

### 15.2 必須掲載情報

- 投票日
- 選挙名
- 対象地域
- 選挙種別
- 公式ページ
- 出典
- 最終確認日時

### 15.3 掲載ポリシー

- 候補者を恣意的に評価しない
- 事実と要約を分ける
- 公報や公式説明がある場合は原典リンクを最優先にする

## 16. データ運用前提

### 16.1 主要データソース

- 日本郵便の郵便番号データ
- 総務省、統計局、e-Stat
- 都道府県・市区町村の選挙管理委員会サイト

### 16.2 運用上の現実

- 候補者、公報、投票所は PDF や自治体個別ページに分散しやすい
- 完全自動化は初期から前提にしない
- 機械取得 + 手動確認のハイブリッド運用を前提にする
- そのため、MVP では完全網羅よりも、確認済み情報の正確さを優先する

## 17. 固定した意思決定

この PRD では、以下を固定した。

1. MVP の入力は郵便番号を基本とする
2. MVP の主価値は「次の選挙が分かること」である
3. MVP の必須ページは、トップ、地域、個別選挙、全国スケジュール、投票率データである
4. 候補者、公報、投票所、期日前投票は「確認済みのものだけ」を段階的に載せる
5. ニュース量や評論ではなく、中立なナビゲーションで勝つ
6. SEO を考え、地域ページと個別選挙ページは URL を持つ前提で設計する
7. 出典、確認日時、確認状態は必須項目とする
8. 郵便番号が曖昧な場合は、推測で自治体を決め打ちしない
9. 市区町村選挙と補欠選挙は、全国完全網羅ではなく確認済みから段階的に広げる

## 18. リスク

- 地方選挙データの取得難易度が高い
- 自治体ごとに形式がばらつき、更新が不規則
- 候補者情報や投票所情報の正規化コストが高い
- 実装よりもデータ運用がボトルネックになる可能性が高い

## 19. MVP 完了条件

以下を満たしたら MVP 完了とみなす。

1. ユーザーが自分の地域の次の選挙を見つけられる
2. 個別選挙ページで基本情報と公式リンクが見られる
3. 出典、確認日時、確認状態を表示できる
4. 全国の選挙スケジュールを一覧で見られる
5. 投票率などの基礎データを確認できる
6. データ未整備の項目を未整備と明示できる
7. 過去の選挙データを現役情報のように誤表示しない
8. 郵便番号が曖昧な場合に、誤った自治体を断定表示しない
9. 公開中の全選挙ページが鮮度ルールに違反していない

## 20. 先送りした論点

以下は重要だが、MVP 着手の前提条件にはしない。

- 郵便番号と自治体の正規化をどの粒度で持つか
