# サイト全体 充実度レビュー

作成日: 2026-04-28  
対象: `site/` の公開静的サイト、`data/v1/` の canonical data、現行生成物

## 総評

現状は「公式情報への案内サイト」としての骨格はできている。トップページの選挙検索、公式リンク導線、検証方針、現職首長台帳、都道府県議会系台帳は、すでに運用可能な水準に近い。

一方で、当初の中心価値である「生活者が自分の地域の次の選挙に迷わず到達する」体験はまだ完成していない。郵便番号入口は実装上は3桁 prefix の補助検索であり、明示的な入力導線や全国的な住所解決ではない。地域別ページと通常選挙ごとの個別ページも不足しているため、MVP完成というより「データ基盤とトップページ検索の初期版」と見るのが妥当。

## 強い点

### 1. 公式ソース中心の方針が守られている

`data/v1/election_resource_links` の401件はすべて `verified` で、非公式リンクは混在していない。候補予定者の観測情報を official resource と分けるルールも運用上重要で、現時点では守られている。

### 2. トップページの基本導線は成立している

「選挙を探す」から、キーワード、地方区分、都道府県、市区町村で選挙を絞り込める。詳細パネルには投票日、告示日、公式サイト、リソースリンクがまとまっており、最低限の案内サイトとして機能している。

### 3. 台帳系コンテンツの厚みがある

現職市長792件、現職知事47件、都道府県議会系データの整備は、単なる選挙日程サイトより強い。将来的に「自治体・議会・選挙の横断案内」に広げられる土台がある。

### 4. 監査・runbook が蓄積され始めている

市長選リンク収集、公式リンク確認、議会任期確認などの runbook と audit が残っている。属人的な作業から、再実行可能な運用へ移せる状態になりつつある。

## 不足・リスク

### P0: 「自分の選挙」入口がまだ完成していない

郵便番号検索は `site/assets/app.js` に実装されているが、3桁 prefix による補助マッチであり、7桁郵便番号から市区町村を確定する仕組みではない。UI上も「郵便番号で探す」専用入口としては目立っていない。

この状態で「郵便番号を入れれば自分の選挙が分かる」と強く打ち出すと過大表示になる。現段階では「地域名・自治体名・郵便番号の一部でも探せる」程度の位置づけが適切。

### P0: レガシーページが公開品質を下げている

`site/pages/schedule.html` と `site/pages/turnout.html` は、現在の `site-data.js` ではなく transitional な `data/elections.js` を参照している。フッターにも `href="#"` が残っており、現行トップページとの統一感が弱い。

特に投票率ページは、表示内容に対して出典・確認日・算出方法がページ上で十分に見えない。政治・行政情報サイトとしては、ここは信頼性リスクになる。

### P0: 公開ページ内に小さな品質問題がある

- `site/pages/current-mayors.html` に `href="市長プロフィール"` という不正なリンクが残っている。
- `site/pages/logo-mark-review.html` と `site/pages/logo-mark-soft-review.html` は、公開サイト配下よりも `artifacts/private/` や内部資料側に置くべき内容に見える。

### P1: 公式リソースの時期別管理がまだ弱い

今後選挙54件はすべて何らかのリソースを持っているが、候補者一覧・選挙公報・投票所のリンクは選挙種別や時期で差が大きい。

- 今後の市長選35件のうち、投票所リンクあり: 21件
- 今後の知事選2件のうち、投票所リンクあり: 1件
- 今後の議会選17件のうち、投票所リンクあり: 3件
- 2026年5月から8月までの市長選26件のうち、投票所リンクあり: 21件

候補者一覧と選挙公報は、告示前に未掲載なのは自然。ただし「未確認」なのか「公式未掲載を確認済み」なのかをデータや監査で区別しないと、運用上の再確認漏れが起きる。

### P1: 統計の基準日が分かりにくい

生成済み `site-data.js` の `stats.byPhase.upcoming` は69件だが、2026-04-28時点で `voteDate >= 2026-04-28` として数えると54件だった。これは生成時基準日や phase 計算ロジックの違いかもしれないが、公開上の「今後」表示に関わるため確認が必要。

### P2: 個別ページと地域ページが薄い

通常の選挙詳細はトップページの詳細パネルで表示されるが、独立した個別選挙ページは3件しかない。地域ページもまだ主要導線として整っていない。

検索結果から公式リンクへ飛べるだけでは、SNS共有、検索流入、地域別の説明、告示後の更新履歴を受け止めにくい。

## 次に進めること

### 1. 公開品質・信頼性クリーンアップ

最優先。外部調査より先に、現行サイトの信頼性を落としている小さな問題を直す。

- `current-mayors.html` の不正リンクを修正する。
- `schedule.html` と `turnout.html` のフッター、ナビ、データ参照を現行構成に寄せる。
- 投票率ページに出典、確認日、対象選挙、算出方法を表示する。
- logo review ページを公開導線から外すか、内部資料へ移す。
- `stats.byPhase.upcoming` の基準日を確認し、表示文言を調整する。

### 2. 「自分の選挙」入口をMVP水準にする

サイトの価値に直結するため、次の大きな改善はここ。

- トップページに明示的な郵便番号・自治体名入力欄を作る。
- 現状の3桁 prefix 検索を過大表示しない文言にする。
- 7桁郵便番号から自治体を確定するか、まずは自治体選択フローを強化する。
- 地方区分、都道府県、市区町村の階層を使って迷わず絞り込めるようにする。

### 3. 告示後再確認フローをカレンダー化する

9月以降の市長選リンク収集は今は止める。次に必要なのは、近い選挙の告示後に公式公開される候補者一覧・選挙公報を取り逃がさない仕組み。

- 5月・6月の選挙を優先する。
- 告示日の翌営業日を再確認日として記録する。
- `candidate_list` と `bulletin` について、未掲載確認済み・掲載確認済み・再確認予定を区別する。
- 内部監査に「次に確認する選挙」を出すスクリプトを追加する。

### 4. 地域ページと個別選挙ページを生成する

検索だけでなく、共有・検索流入・地域別案内に耐える構造へ進める。

- 都道府県ページを生成する。
- 市区町村ページを生成する。
- 選挙ごとの個別ページをテンプレート生成する。
- 各ページに公式リンク、確認日、更新履歴を出す。

### 5. 議会ページの標準化

議会ページは量がある一方で、更新基準や出典表示の統一が次の課題。

- 30件の議会ページに共通メタ情報を付ける。
- 最終確認日、主要ソース、次回確認予定を表示する。
- 都道府県議会公式リンク台帳との相互リンクを強める。

## 推奨順序

1. 公開品質・信頼性クリーンアップ
2. 「自分の選挙」入口のMVP化
3. 5月・6月選挙の告示後再確認カレンダー
4. 地域ページ・個別選挙ページの生成
5. 議会ページ標準化

この順番がよい理由は、先に公開品質を整えると、次の機能追加や調査結果を安心して載せられるため。9月以降の市長選収集よりも、現行公開面の信頼性と近い選挙の再確認フローを優先すべき。
