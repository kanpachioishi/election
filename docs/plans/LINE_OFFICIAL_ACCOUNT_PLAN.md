# LINE Official Account Plan

最終更新: 2026-04-18 JST

> Deprecated: 現在の公開通知導線は LINE ではなく PWA / Web Push です。この文書は旧構想の記録として保持しています。

## 1. 背景

この構想の出発点は、

- 選挙が近いことを急に知った
- 十分な情報を集める時間がなかった
- もっと早く気づけていれば判断材料を持てた

という悔しさにある。

このプロジェクトは、公式情報への導線を整理することに加えて、
「知らないうちに大事な選挙が近づいていた」を減らすことを目指す。

そのための将来案として、公式 LINE アカウントを持つ。

## 2. 目的

LINE で実現したいことは、ニュース配信ではなく選挙の気づき支援である。

目標:

- 住んでいる地域や関心地域の選挙が近づいたことを早めに知らせる
- 候補者一覧、選挙公報、期日前投票、投票所案内などの公式情報にすぐ届ける
- 「気づくのが遅かった」を減らす
- 公式情報を見るまでの手間を減らす

## 3. 非目的

少なくとも初期段階では、次のことはやらない。

- 政治的な立場を推す配信
- 候補者評価や投票先の推奨
- 雑多な政治ニュースの配信
- 長文記事中心のメディア運用

まずは、選挙の存在と公式導線を届けることに集中する。

## 4. 想定する通知

優先度順に並べると、まず欲しいのは次の通知。

### 4.1 選挙が近い通知

- 例: `あなたの地域の市長選まであと30日`
- 目的: 早めの気づき
- 必要データ:
  - 対象地域
  - 投票日
  - 選挙種別

### 4.2 告示通知

- 例: `本日告示。候補者情報と選挙公報の確認先はこちら`
- 目的: 情報収集の開始点を作る
- 必要データ:
  - 告示日
  - 候補者一覧の公式リンク
  - 選挙公報リンク

### 4.3 期日前投票開始通知

- 例: `期日前投票が始まりました`
- 目的: 投票機会を逃しにくくする
- 必要データ:
  - 期日前投票ページ
  - 期日前期間
  - 投票所案内

### 4.4 投票日前日・当日通知

- 例: `明日は投票日`
- 目的: 最後のリマインド
- 必要データ:
  - 投票日
  - 投票所案内
  - 投票時間

### 4.5 公式資料公開通知

- 例: `選挙公報が公開されました`
- 目的: 見るべき資料が出たことを伝える
- 必要データ:
  - 選挙公報 URL
  - 候補者一覧 URL
  - 投票所 URL

## 5. 初期の配信設計

最初から細かい個別最適を目指しすぎない。

初期案:

- 配信対象は `都道府県知事選` と `主要な市長選` から始める
- 通知は `あと30日 / 告示日 / 期日前開始 / 前日` の4本を基本にする
- まずは手動または半手動で送れる設計にする
- ユーザーごとに `1つの市区町村` を登録できるようにする
- 登録した市区町村の親である `都道府県` も受信対象に含める
- 配信は `登録地域に一致した人だけ` に送る

理由:

- 今の dataset でも比較的作りやすい
- 精度確認をしながら運用を学べる
- 送るべき通知の価値が見えやすい

## 6. LINE に必要なデータ

今のプロジェクト資産と照らすと、使えるものと足りないものがある。

すでに使いやすいもの:

- `data/v1/elections.json`
  - 選挙名
  - 種別
  - 告示日
  - 投票日
  - 地域
- `data/v1/election_resource_links`
  - 候補者一覧
  - 選挙公報
  - 期日前投票
  - 投票所案内

まだ整備が必要なもの:

- どの通知をいつ送るかのルール定義
- 地域と配信対象の紐づけ
- 通知に使う短いテンプレ文
- 公式情報公開の検知方法
- `この選挙は LINE 通知対象にするか` のフラグ

## 7. 実装前に決めること

実装前に最低限決めたい論点は次の通り。

### 7.1 誰に送るか

- 全国一律配信にするか
- 都道府県単位で受け取り地域を選べるようにするか
- 将来的に市区町村単位まで絞るか

このプロジェクトの方針としては、初期から `1ユーザー1市区町村登録` に寄せる。

理由:

- ユーザーに不要な通知を送らないことを優先したい
- `県の情報` と `市の情報` を分けて扱いやすい
- 「登録したところ以外の情報はいかない」を設計として明確にできる

### 7.2 何を対象にするか

- 知事選のみ
- 知事選 + 一部の市長選
- 全国の首長選

初期は `知事選 + 優先度の高い市長選` が無理がない。

### 7.3 どこまで自動化するか

- 完全手動
- 通知候補だけ自動生成
- 自動送信まで行う

初期は `通知候補の自動生成 + 最終確認は手動` が安全。

## 8. 段階的な進め方

### Phase 1: 構想固定

- 送りたい通知の種類を固定する
- 対象選挙の範囲を固定する
- 地域選択の単位を決める

### Phase 2: データ準備

- `LINE通知対象` の考え方を決める
- 配信に必要な URL の優先順位を決める
- 不足リンクの棚卸しをする

### Phase 3: 運用試作

- 手動で通知文を作る
- 小さく試す
- 何の通知が役立つかを確認する

### Phase 4: 実装

- 配信対象管理
- 通知候補生成
- LINE 配信導線

## 9. 今の時点での結論

この構想は、今の dataset 拡張と相性が良い。

今やっていること:

- 選挙の existence を持つ
- 告示日と投票日を持つ
- 公式リンクを持つ

これはそのまま、将来の LINE 通知の土台になる。

したがって、今は LINE 実装を急がず、

- `elections.json` の精度を上げる
- `election_resource_links` を厚くする
- 通知に向く選挙を見分ける

ことを続ければよい。

## 10. 次に着手するときの最小タスク

次回この構想を実装寄りに進めるときは、まず次の 3 つから始める。

1. `LINE で送る通知種類` を 4 本程度に固定する
2. `県配信` と `市配信` の振り分けルールを決める
3. `通知に必要なデータ項目一覧` と `ユーザー登録項目` を docs 化する

この 3 つが決まれば、実装の議論に入りやすい。

## 11. 公式LINEの作り方

2026-04-18 時点では、Messaging APIチャネルを先に作るのではなく、
次の順番で進めるのが公式手順である。

1. LINE Business ID を用意する
2. LINE公式アカウントを作成する
3. LINE Official Account Manager で Messaging API を有効にする
4. LINE Developersコンソールでチャネル設定を行う
5. Webhook URL、チャネルアクセストークン、チャネルシークレットを設定する

補足:

- 2024-09-04 以降、Messaging APIチャネルを LINE Developersコンソールから直接新規作成する方式は終了している
- Provider はあとで付け替えできないので、既存のLINE Loginや将来の連携先と同じProviderに載せるかを最初に決める

実務上の最小ステップ:

1. LINE Business ID で LINE Official Account Manager にログインする
2. 公式アカウントの名前、業種、用途を入力してアカウントを作る
3. Manager 側で Messaging API 利用開始を押す
4. LINE Developersコンソールに入り、作成された Messaging APIチャネルを確認する
5. Webhook URL を設定し、疎通確認を行う
6. チャネルアクセストークンを発行する
7. 友だち追加用の QR コードや導線をサイトに置く

## 12. この案件での推奨運用方式

### 12.1 配信の前提

この案件では、`一斉配信` を前提にしない。

やりたいこと:

- 各人が `1つの市区町村` を登録する
- その人には `登録した市区町村` と、その親である `都道府県` の情報を送る
- 登録していない都道府県、市区町村の情報は送らない

例:

- `鹿児島県 / 姶良市` を登録した人には、鹿児島県知事選などの県情報と、姶良市長選などの市情報を送る
- `鹿児島県 / 姶良市` 登録者に、`霧島市` や `宮崎県` の通知は送らない

### 12.2 推奨アーキテクチャ

初期は次の構成が最も扱いやすい。

- LINE公式アカウント
- Messaging API
- Webhook を受けるボットサーバー
- ユーザー登録情報を持つDB
- 配信対象を計算して push / multicast する配信ジョブ

登録時に保持する最小情報:

- `line_uid`
- `pref_region_id`
- `municipality_region_id`
- `follow_status`
- `delivery_status`
- `updated_at`

配信ルール:

- `県の通知` は `pref_region_id` が一致するユーザーだけに送る
- `市の通知` は `municipality_region_id` が一致するユーザーだけに送る
- 全友だち向け broadcast は使わない

### 12.3 登録フロー

最初の実装としては、次の流れが現実的。

1. ユーザーが友だち追加する
2. `follow` webhook で `userId` を取得する
3. 最初の応答で `地域を登録してください` と案内する
4. LIFF で本人確認し、LINE の userId と保存先ユーザーを結びつける
5. LIFF の登録画面で `都道府県 -> 市区町村` を選ばせる
6. 選択結果をDBに保存する
7. `登録内容を変更する` 導線も常設する

操作UIの候補:

- 主経路: リッチメニューから LIFF を開く
- 補助導線: postback で LIFF を起動する
- LIFF では既存サイトの地域選択UIを流用する

初期は `LIFF を主経路` にするのが最も現実的。

### 12.4 配信方式

この案件では、まず `push message` を基本に考える。

理由:

- `登録した地域だけに送る` という条件を、自前DBの条件式でそのまま実装できる
- 誰に送ったかを監査しやすい
- 市単位の小さい配信でも扱いやすい

`multicast message` は、同じ内容を同時に複数人へ送るときの最適化として使える。

`narrowcast` は将来の選択肢にはなるが、初期の主軸にはしない。

理由:

- この案件では LINE 側の属性推定より `ユーザーが明示登録した地域` を信頼したい
- 配信条件の説明責任を自前DBで持ったほうがわかりやすい
- 実装初期は `誰に届くか` を完全に追いやすい形がよい

## 13. セキュリティ記録

### 13.1 管理画面と権限

- LINE Business ID は共有アカウントにしない
- 管理者は最小人数に絞る
- 開発者には必要な範囲だけ権限を付与する
- チャネルシークレットやMessaging API設定を触れるのは Admin に限定する
- 退職、委託終了、担当変更があったら権限を即日棚卸しする

### 13.2 認証

- LINE Business ID の二段階認証を前提運用にする
- ログイン用メールアドレスは共有メーリングリストではなく、責任者単位で分ける
- LINEログイン連携を使う場合も、誰の端末で承認されるのかを明確にする

### 13.3 Webhook

- Webhook は必ず HTTPS にする
- Webhook 受信時は署名検証を通るまで処理しない
- IP 制限だけに頼らない
- 検証前に request body を加工しない
- Webhook は同期で重い処理をせず、受信後すぐ 200 を返して非同期処理に逃がす
- 失敗した Webhook は再試行キューや dead-letter を通して回復できるようにする

### 13.4 トークンと秘密情報

- チャネルアクセストークン、チャネルシークレットは環境変数やシークレットマネージャで管理する
- リポジトリに平文で置かない
- 画面共有、スクリーンショット、ログ出力で漏らさない
- 漏えいが疑われたら、アクセストークン失効 / 再発行、必要ならチャネルシークレット再発行を行う

### 13.5 ユーザーデータ

- 保持する個人情報は最小限にする
- 初期段階では `LINE userId` と `都道府県コード` と `市区町村コード` を中心にし、住所全文は持たない
- 配信対象抽出に不要なプロフィール情報は保存しない
- DB上では `誰がどの地域を登録したか` を見られる人を絞る
- 退会、ブロック、配信停止の導線を明示する

### 13.6 誤配信対策

- `県向け通知` と `市向け通知` で配信SQLや抽出条件を分ける
- 本番送信前に `配信対象件数` と `対象地域名` を必ずプレビューする
- `1人のテストユーザー`、`1市だけのテスト`、`1県だけのテスト` を先に通す
- 本番の broadcast API は使わない、または権限的に使えない運用にする
- 配信ログに `message_id`、`election_id`、`target_scope`、`target_region`、`recipient_count` を残す

### 13.7 初期実装の安全側ルール

初期は次の制約を置くと安全。

- 自動送信ではなく `通知候補生成 + 人手承認 + 送信` から始める
- 通知テンプレートは固定文に近づける
- 1回の送信ごとに対象地域を1つに限定する
- `県配信` と `市配信` を同時に混ぜない
- 登録変更はいつでも上書き可能にする
- 承認時に見た対象者集合を固定し、送信時に勝手に差し替えない
- 本番では作成者と承認者を分ける

## 14. 今回の要件に対する結論

今回の要件:

- 各人が市を登録する
- 県と市の情報を受け取れるようにする
- 登録したところ以外の情報はいかないようにする
- 一斉配信ではない

この要件に対する実装方針は、次で固定するのがよい。

- ユーザーごとに `1つの市区町村` を正本として保存し、その親 `都道府県` も保持する
- 通知対象は自前DBで抽出する
- 送信は `push` を基本、必要なら `multicast` で最適化する
- `broadcast` は使わない
- 初期は `手動承認あり` で運用する
