## 1. この文書の目的

この文書は、MVP を実装できる粒度まで分解し、次の3点を固定するためのものです。

- 何から着手するか
- どのタスクが何に依存するか
- どの状態になれば次へ進めるか

ここでいうタスク分解は、単なる ToDo の列挙ではありません。  
「公開してよい状態を守れる順番」で並べることを優先します。

## 2. タスク分解の原則

- 画面より先に公開基盤を作る
- `verified` 以外の情報を主要表示に使わない
- 郵便番号が曖昧なときは自治体を決め打ちしない
- まずは1本の縦切りで `収集 -> 正規化 -> 確認 -> 公開` を通す
- 独立画面は増やしすぎず、MVP の 5 画面に集中する
- 公開前監査を通らない限りリリースしない

## 3. MVP の着手順

MVP は次の順番で進めるのが最も安全です。

1. 公開ルールを固定する
2. 最小データ基盤を作る
3. 郵便番号解決を作る
4. 公開前監査を作る
5. 代表地域で縦切りを通す
6. 5画面を接続する
7. 再確認・失効・アーカイブを整える
8. リリース判定を行う

この順番にしている理由は、トップページや地域ページを先に作っても、`verified` 判定や郵便番号の分岐がなければ、正しい情報を安全に出せないからです。

## 4. 優先度の定義

- `P0`: これがないと MVP の価値か安全性が成立しない
- `P1`: MVP 公開時にあったほうがよいが、P0 完了後に着手してよい
- `P2`: MVP 公開判定の最後に行う、または公開直前に詰める

## 6. 実装順のまとまり

### Step 0. 公開ルールを固定する

対象ソース、対象選挙、除外範囲、掲載条件を先に決めます。  
ここで曖昧さを残すと、後ろの実装がすべて揺れます。

対応タスク:

- T01
- T07

### Step 1. 最小データ基盤を作る

MVP はデータ構造の勝負です。  
先にモデルと検証メタデータを固めて、後続画面の判断材料を作ります。

対応タスク:

- T02
- T03
- T04

### Step 2. 郵便番号解決と公開ブロックを作る

このサービスの核は `あなたの地域` に正しく辿り着けることです。  
同時に、誤った公開を止める仕組みもここで入れます。

対応タスク:

- T05
- T06

### Step 3. 代表地域で縦切りを通す

全国対応を広げる前に、1 地域で最後まで通ることを確認します。  
この段階で、データ投入と確認運用の現実的な重さも分かります。

対応タスク:

- T08
- T09
- T10
- T11
- T12

### Step 4. 横展開できる画面を作る

縦切りが通ったあとで、一覧系とデータ系のページへ広げます。  
ここを先に作ると、基礎の弱さを UI で隠す形になりやすいです。

対応タスク:

- T13
- T14

### Step 5. 再確認とアーカイブを整える

公開できることより、公開し続けられることを重視します。  
MVP でも、古い情報を最新のように見せないラインは越えないようにします。

対応タスク:

- T15
- T16

### Step 6. リリース判定を行う

最後は主観ではなく、監査項目と表示要件で公開可否を決めます。

対応タスク:

- T17

## 7. 並列で進められるもの

次の組み合わせは、依存を満たせば並列で進めやすいです。

- T02 と T07
- T09 と T10
- T13 と T14
- T15 と T16

ただし、次のものは並列化しないほうが安全です。

- T05 より前にトップページ導線を作ること
- T06 より前に公開判定をすること
- T08 を飛ばして全国ページを先に広げること

## 8. 今やらないこと

MVP では次をタスク化しません。

- 候補者比較ページ
- 候補者詳細ページ
- 期日前投票の独立ページ
- 投票所案内の独立ページ
- 独立 FAQ ページ
- 通知機能
- アカウント機能
- お気に入り機能
- コメントや参加型機能

これらは価値が低いからではなく、MVP の信頼性と運用可能性を優先するために後ろへ送ります。

## 9. MVP 公開の最低条件

次の条件が揃ったら、MVP の公開判定に入れます。

- 郵便番号入力が `resolved / ambiguous / unresolved` で正しく分岐する
- トップページ、地域ページ、個別選挙ページ、全国スケジュール一覧ページ、投票率ページの 5 画面がつながっている
- 公開情報に `source_url`、`confirmed_at`、`status` が表示されている
- `verified` 以外の情報を主要表示として見せていない
- 監査で、未解決郵便番号、必須項目欠損、未カバー地域、`needs_update` 件数を確認できる
- `ended` と `archived` の表示ルールがあり、古い情報を最新のように見せない

## 10. 最初の実装スプリント

最初のスプリントは、次の 6 タスクに絞るのがよいです。

1. T01 公式ソース一覧とカバー方針を固定する
2. T02 最小データ構造を実体化する
3. T03 検証メタデータの必須化を行う
4. T05 郵便番号解決フローを作る
5. T06 公開前監査の基盤を作る
6. T08 代表地域のデータ収集を 1 本通す

この 6 つが通ると、ようやく「安全に画面を作り始めてよい」状態になります。
