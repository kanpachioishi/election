# Politics Expansion Concept

更新日: 2026-04-19

## 1. 結論

このサイトは、`選挙の公式情報への案内サイト` から、`市民が政治の動きを追うための案内サイト` へ少しずつ広げる。

ただし、いきなり何でも扱う総合政治メディアにはしない。
まずは次の 3 本柱で整理する。

1. `選挙の公式情報案内`
2. `議会・行政の要点整理`
3. `初学者向けの見方ガイド`

この順番にする理由:

- いまの強みは、選挙ごとの公式情報導線にある
- AdSense 的にも、単なるリンク集では独自性を厚くしにくい
- ユーザー価値としては、選挙だけでなく「ふだんの政治の流れ」が見える方が、関心を持ち続けやすい
- ただし、広げすぎると何のサイトか分からなくなるので、`選挙を核にして周辺を広げる` のが安全

## 2. 新しいコンセプト案

### コンセプト文

このサイトは、選挙のときだけでなく、ふだんの議会や行政の動きも含めて、
`市民が公式情報へ迷わずたどり着き、要点をつかみ、次の行動につなげるための案内サイト`
とする。

### ひとことで言うなら

`政治を判断するサイトではなく、政治を追えるようにするサイト`

### やること

- 公式情報の入口を整理する
- 情報が散らばる行政・議会の内容を、原文に進みやすい形で要約する
- 初めての人にも「どこから見ればいいか」が分かるようにする

### やらないこと

- 候補者や会派の支持・不支持を誘導する
- 感情的な煽り見出しで関心を取る
- 原文なしの断定的な要約を出す
- 出典不明の政治情報を載せる

## 3. サイトの役割の再定義

### これまで

- 選挙ごとの候補者一覧
- 選挙公報
- 期日前投票
- 投票所案内
- 公式リンク導線

### これから

上の役割はそのまま残しつつ、次を足す。

- `議会で何が決まったか`
- `何がまだ審議中か`
- `暮らしへの影響がどこにあるか`
- `原文を読むならどこを見るか`

つまり、`投票の日のためのサイト` から、`投票までと投票のあいだをつなぐサイト` に広げる。

## 4. 議会要約ページの最小構成

最初の試作は、1 自治体・1 会議・1 本で十分。
大きくしすぎず、まずは「読める」「原文へ戻れる」「誤解しにくい」の 3 条件を満たす。

### 必須ブロック

1. `ページ概要`
   - 自治体名
   - 議会名 / 委員会名
   - 会議日
   - 確認日
   - 出典リンク

2. `3行要約`
   - この会議で大きかった論点
   - 決まったこと / 継続審議 / 報告事項のどれか
   - 暮らしへの影響の有無

3. `何が決まったか`
   - 可決 / 否決 / 継続 / 報告
   - 条例 / 予算 / 契約 / 意見書などの区分

4. `何が議論になったか`
   - 主要論点を 2-4 項目
   - 賛否や論点の違いを、断定しすぎず整理

5. `暮らしへの影響`
   - 市民から見て、何が変わる可能性があるか
   - 影響が不明なら `現時点では大きな影響は読み取りにくい` と書く

6. `原文・公式情報`
   - 会議録
   - 議案一覧
   - 配信動画 / 録画
   - 自治体の該当ページ

7. `この要約の位置づけ`
   - このページは原文の代わりではない
   - 最終確認は公式資料で行う
   - 解釈が難しい箇所は原文リンクを見る

### 最小テンプレート例

```text
タイトル: 2026年4月 小城市議会 臨時会の要点

概要:
- 自治体: 小城市
- 会議: 2026年4月臨時会
- 会議日: 2026-04-XX
- 確認日: 2026-04-XX
- 出典: 会議録 / 議案一覧 / 録画

3行要約:
- 今回は ○○ と ○○ が主要論点でした。
- △△ は可決、□□ は継続審議でした。
- 暮らしへの直接影響は ○○ に出る可能性があります。

何が決まったか
何が議論になったか
暮らしへの影響
原文リンク
注意書き
```

## 5. 議会要約を書くときのルール

### 事実と解釈を混ぜない

- `可決された`
- `質疑があった`
- `賛成多数だった`

は事実。

- `重要法案だった`
- `市民にとって大きい`

は解釈。

解釈を書くなら、`なぜそう言えるか` を短く添える。

### 原文に戻れるようにする

要約だけ読んで終わらず、必ず公式ページへ戻れる構造にする。
AdSense 的にも、ユーザー価値的にも、これがサイトの信頼の核になる。

### 煽らない

議会コンテンツは興味を引くために強い言葉を使いたくなるが、
このサイトの役割は `関心を持ってもらうための入口` であり、`怒らせることで滞在を伸ばすこと` ではない。

### 主張の代理をしない

候補者、会派、首長、議員の意見を紹介することはあっても、
サイト自身が代わりに結論を出す構造にはしない。

## 6. 選挙と議会をどう共存させるか

### 基本方針

選挙と議会は別カテゴリに見えて、利用者にとっては一本の流れである。

- 選挙で選ぶ
- 議会や行政で決まる
- 次の選挙で見直す

この循環を見せることが、このサイトの独自価値になる。

### 情報設計

トップでは主役をまだ `選挙を探す` に置く。
ただし、説明や導線で `議会の要点` へ自然につなげる。

おすすめの構造:

1. トップヒーロー
   - 公式情報への案内
   - 通知設定

2. 説明セクション
   - このサイト独自の価値
   - 確認と更新の方針
   - 利用前に知っておいてほしいこと

3. 次の導線
   - `選挙を探す`
   - `議会の要点を見る`
   - `このサイトについて`

### コンテンツ上の役割分担

#### 選挙ページ

- 直近の行動に必要な情報
- 投票前の確認
- 候補者一覧 / 公報 / 投票所案内

#### 議会ページ

- 投票後の政治を追う情報
- 議案 / 会議 / 予算 / 委員会
- 暮らしへの影響の整理

#### 初学者向けページ

- 議会とは何か
- 首長と議会の違い
- 予算や条例を見るときの入口

## 7. 最初にやるべき最小実装

いきなり全国展開しない。
まずは 1 自治体の試作でよい。

### Step 1

- `議会要約ページ` を 1 本作る
- 出典は会議録 / 議案一覧 / 録画の 3 系統を基本にする
- トップに `議会の要点を見る` の仮導線を入れる

### Step 2

- ページテンプレートを固める
- `この要約の位置づけ` と `原文リンク` を定型化する
- 読みやすい要約長の目安を固める

### Step 3

- 対象自治体を 2-3 に増やす
- 選挙ページとの相互リンクを足す
- 通知との接続を検討する

## 8. 通知との相性

通知は選挙だけでなく、将来的には議会要約とも相性がいい。
ただし、最初から両方通知すると重くなるので、段階を分ける。

### まずやる通知

- 告示日
- 期日前投票開始
- 投票日前日
- 投票日当日

### 将来やる通知

- `今月の市議会で決まったこと`
- `予算の要点`
- `暮らしに影響する条例改正`

### 注意

議会通知は選挙通知よりも頻度が高くなりやすい。
そのため、選挙通知と同じ扱いではなく、`受け取りたい人だけ別で登録` が向いている。

## 9. この方向で得られるもの

- AdSense 向けの独自性が出しやすい
- 単なるリンク集から抜け出せる
- 選挙だけで終わらず、政治への継続関心につなげられる
- サイトのポリシーと運営思想が一致しやすい

## 10. いまの結論

結論としては、`選挙だけに閉じない` 方向はかなり正しい。

ただし、`政治全般サイト` に一気に広げるのではなく、
`選挙を核に、議会の要点整理を次の柱として足す`
のが一番自然で、いまの資産とも相性が良い。

次にやるべき実務はこれ:

1. 議会要約ページの試作を 1 本作る
2. トップ導線で `議会の要点` を 1 か所見せる
3. 要約テンプレートと注意書きを固定する
