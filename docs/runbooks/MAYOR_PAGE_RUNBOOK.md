# Mayor Page Runbook

最終更新: 2026-04-17 JST

このメモは、市長選の専用ページを追加するときの運用ルールをまとめたものです。
藤岡市長選ページで見えた「知事選より情報が薄い」「公式候補者一覧が遅い」「自治体の案内ページ依存が強い」という前提を、次の市長選へ横展開しやすい形にしています。

共通ルールは `docs/GOVERNOR_PAGE_RUNBOOK.md` を土台にしつつ、この runbook では市長選特有の差分を優先して書いています。

## 1. いつ専用ページを作るか

- 市長選は原則としてメインページ掲載までとし、専用ページは大都市など例外的に優先度が高い場合だけ作る。
- メインページの公式リンク一覧だけでは足りず、候補予定者のプロフィールや出馬動向を1ページにまとめたいとき
- 告示前で、公式候補者一覧はまだ無いが、現職情報・議会情報・本人発信・報道を組み合わせれば最低限のページが成立するとき
- 候補者数は少なくても、「誰が出そうか」「どこに公式情報があるか」を1ページで見たい需要が強いとき

逆に、次の状態では専用ページを急がない。

- 候補予定者の確認がほぼ進んでいない
- 選挙日程の公式案内ページすら固まっていない
- 現職や主要人物の基礎プロフィール導線がまったく拾えていない

この段階では、まず `election_resource_links` の公式導線を固める。

## 2. 市長選ページの前提

- 市長選は、知事選より公示前情報が薄いことが多い
- 公式候補者一覧が出る前は、候補者比較より「候補予定者が誰か」「最低限どこまで確認できたか」を見せるページになりやすい
- 現職がいる場合は、自治体の `市長の部屋`, `市長プロフィール`, `市長コラム`, `市長トピックス` などが主要導線になる
- ただし、それらは「現職としての公務プロフィール」であり、選挙用の公式サイトとは限らない
- 報道の1本が切れると根拠ごと弱くなるので、知事選以上にリンク運用を厳密にする

## 3. 追加するファイル

最低限、次の 6 つをそろえる。

1. `data/v1/elections.json`
2. `data/v1/election_resource_links/{election_id}.json`
3. `data/v1/candidate_signals/{election_id}.json`
4. `data/v1/candidate_profiles/{election_id}.json`
5. `site/elections/{slug}.html`
6. `site/assets/{slug}.js`

必要に応じて追加するもの:

- `data/v1/candidate_endorsements/{election_id}.json`
- `site/assets/{slug}.css`
- `site/assets/app.js` の専用ページ導線

市長選は知事選より独自事情が出やすいが、まずは直近の市長選ページを複製して、`ELECTION_ID` と文言を差し替えるのが早い。

## 4. データの基本ルール

### 4.1 `elections.json`

- `page_status` を必ず入れる
- 告示前は `official_candidate_list_status = "not_included"` を使う
- `page_updates` に、専用ページを追加した日付と要約を残す
- `verification.source_url` には、投票日・告示日を確認した市選管のページか公式 PDF を入れる
- `description` は簡潔にとどめ、候補状況の断定はここに書きすぎない

### 4.2 `candidate_signals`

- 候補予定者・出馬動向はここに入れる
- 市長選は根拠が 1 件しか取れない場面があるので、その場合は summary を弱めて書く
- 強い根拠が報道1本だけなら、`confidence = medium` も積極的に使う
- 議会の一般質問通告一覧や録画配信ページは、出馬表明そのものを直接言い切る根拠ではないことがある
- 公式情報が間接根拠しか持たない場合は、「報道ではこうされている」「現時点では報道ベースで扱う」と書いて、確認できた範囲に要約を狭める
- `interested` や `related_interest` 相当の人物は、候補予定者一覧には混ぜない

### 4.3 `candidate_profiles`

- 役割は、評価ではなく「誰なのかを見分けるための事実整理」
- 市長選では、選挙用公式サイトが無いことが珍しくない
- その場合は `市長の部屋`, `自治体プロフィール`, `本人SNS`, `本人ブログ` を現実的な導線として使ってよい
- ただし `official_site_url` に自治体ページを入れるときは、選挙用公式サイトではなく「本人の公的プロフィール導線」として扱っている意識を持つ
- `identity_labels` は `現職市長`, `元副市長`, `市長プロフィールあり`, `市長コラムあり` のような中立ラベルに限る
- 思想、色分け、推薦状況を `identity_labels` に混ぜない

### 4.4 `candidate_endorsements`

- 推薦・支持・擁立は、候補動向と混ぜず別ファイルに分ける
- 市長選では情報がかなり薄いことがあるので、無理に作らなくてよい
- 1 件だけ見つかった段階で断片的にページへ出すより、複数確認できるまで待つ判断もあり

## 5. 公式情報の集め方

市長選で優先する確認先は、次の順。

1. 市選挙管理委員会の選挙案内ページ
2. 投票所一覧、投票所案内図、宣誓書、選挙公報、候補者一覧などの公式 PDF
3. 市議会の一般質問通告一覧、録画配信、会議録
4. 現職市長の公式プロフィール導線
5. 本人発信
6. 報道

知事選との差:

- 都道府県公式ページより、市の個別ページや PDF が主導線になりやすい
- リンクタイトルと実ファイルの中身がズレやすい
- `投票所一覧` と `投票所案内図` のように、似た名称でも別物がある

だから、市長選では「ファイル名がそれっぽい」だけで登録しない。
必ず、公式ページ上の表示文言とカードタイトルを一致させる。

## 6. 候補予定者と関連動向の分け方

- `reported_candidate`, `announced`, `official_candidate` は候補予定者側へ出す
- `related_interest`, `interested` は `関連動向` セクションへ分ける
- 関連動向の人物を、比較帯、候補予定者プロフィール、公式導線比較に混ぜない
- 市長選は「現職が次も出るのでは」という空気のような情報が多いので、明確な根拠のない推測は入れない
- 現職ページが充実していても、出馬根拠が弱ければ候補予定者として断定しない

## 7. プロフィール運用ルール

- まず「この人は誰か」が分かることを優先する
- 市長選では `公式・本人サイト / SNS / プロフィール / 政策` が全部そろわないことを前提にする
- 空欄を埋めるために弱い外部リンクを入れない
- `市長の部屋` や `自治体プロフィール` は、現職を見分ける導線として有効
- 反対に、現職以外の候補者は本人サイトや本人SNSの有無に差が出やすいので、「未掲載」をネガティブに読ませない

肩書確認のルール:

- 現職肩書きは、古いプロフィールだけで断定しない
- 現職市長は自治体公式で確認する
- 元職や議員歴は、直近選挙結果、議会名簿、本人または党の最新プロフィール、報道の順で確認する
- 断定できない場合は `元市議`, `元副市長`, `前市議` のように現在形ではない表現へ逃がす

## 8. 見出しと文言

- 告示前は `候補予定者プロフィール（報道ベース）` を使う
- 公式候補者一覧が公表されたら `公式候補者プロフィール` に切り替える
- ページ冒頭では「公式情報と報道ベースを分けて整理している」と分かるようにする
- 市長選では特に、ページが薄いからといって大げさな見出しにしない
- 公式情報セクションのカード名は、元ページの表示名に寄せる
  例: `投票所一覧` ではなく、元ページが `投票所案内図` ならそのまま使う

## 9. UI の基本ルール

- 主役は候補者名
- ひらがな表記を付ける
- 年齢は名前の横に出すが、名前より目立たせない
- 比較帯は、情報が少ないときほど「あるものだけを静かに出す」
- スロットを埋めるための飾りや、意味の薄い指標は足さない
- 公式情報カードは、市長選では実用性が高いので、投票日・告示日・投票所・宣誓書の導線を壊さない
- 色や記号で思想や立場を想起させる見せ方は避ける
- 内部データでは市長選と議会選を別 record で持ってよいが、利用者向け UI では「同じ日に同じ場所で投票する選挙か」を優先して考える
- 同日・同地域・同投票所の市長選と議会選は、別カードに機械的に分けず「同時選挙」として束ねて見せる
- そのうえで、候補者一覧や候補者比較のように選挙ごとに異なる情報だけを中で分ける

### 9.1 同時選挙の見本

姶良市の `2026-04-19` は、市長選と市議会議員選挙が同日に実施される。
このケースは、今後の同時選挙 UI の見本として扱う。

- 内部データ:
  - `el-mun-46225-mayor-2026`
  - `el-mun-46225-assembly-2026`
  - record は分けて持つ
- 利用者向け表示:
  - メインページでは `姶良市の同時選挙` として 1 つに束ねる
  - 詳細側では `含まれる選挙` として `姶良市長選挙 / 姶良市議会議員選挙` を明示する
- 共通導線:
  - 総合案内
  - 選挙公報
  - 期日前投票
  - 投票所
  - これらは市長選と市議選の共通ページになりうる
- 個別導線:
  - 候補者一覧は市長選と市議選で別 PDF になることがある
  - この場合は `候補者一覧` だけを選挙ごとに分けて見せる
- 判断原則:
  - `別 record だから別カード` にはしない
  - `投票に行く人が、同じ日に一緒に投票するものとして理解するか` を優先する
  - タイトルは公式見出しに寄せる

同じ型が出たら、まず姶良の実装とデータを見本にする。

## 10. メインページとの役割分担

- メインページは公式リンク中心
- 専用ページは、報道ベースの候補予定者、プロフィール、時系列、出典を受け持つ
- 市長選ではメインページとの差が薄くなりやすいので、「誰が出そうか」と「現職がどんな人物か」が分かることを専用ページの主価値にする
- `site/assets/app.js` には、その選挙だけ専用ページへ飛ぶ導線を足す
- 文言は `報道ベースの候補予定者ページへ` など、専用ページの役割が分かるものにする

## 11. 外部リンク運用ルール

- 新しく追加した外部リンクは、公開前に必ず実リンク確認する
- `candidate_signals`, `candidate_profiles`, `candidate_endorsements`, `election_resource_links` の URL は、少なくとも一度はブラウザまたは同等手段で開いて確認する
- リンク先が 404, リダイレクトループ, アクセス不能, 別記事化している場合は、その URL を採用しない
- 転載・ミラー・まとめサイトより、元の発信元を優先する
- 市長選では自治体 PDF の格納ディレクトリ違いが起きやすいので、`group/4` と `group/7` のような差でも必ず実ページから確認する
- カードタイトルとリンク先の実体がズレる場合は、URL だけでなくタイトルと要約も同時に修正する
- リンク先は、可能なら中継の案内ページではなく元の PDF / 元ページへ直リンクする
- ただし、直リンクが不安定、利用条件や注意書きが親ページにしかない、親ページ自体が主要導線である、のいずれかなら中継ページを採用してよい
- HTML ページ自体に投票所一覧、期日前投票所一覧、配架場所、候補者情報などの本文が載っている場合は、そのページを「実体のある元ページ」とみなしてそのまま採用する
- つまり避けるべきなのは「PDF への単なる通過点」であり、「本文に必要情報がある HTML」ではない
- URL を差し替えたら、`last_checked_at` か関連レコードの確認日時も同時に更新する
- URL を追加・修正したら、そのまま終わらせず `node scripts/generate-site-data.mjs` まで実行して、配信用データへ反映する
- 生成後は `site/data/site-data.js` に古い URL が残っていないか確認してから完了にする
- 市長選のリンクを新規に拾ったら、次回の再利用のために `docs/MAYOR_LINK_SOURCE_INVENTORY.md` に「どこを見て見つけたか」を追記する

特に信頼に関わるので、市長選ページの出典リンクは「クリックして開く」確認を省略しない。

## 12. 実装後の確認

最低限、次を通す。

```bash
node scripts/validate-data-v1.mjs
node scripts/generate-site-data.mjs
node --check site/assets/app.js
node --check site/assets/{slug}.js
```

見た目確認も必須:

- PC の実画面スクリーンショット
- モバイルの実画面スクリーンショット
- 公式情報カードと出典リンクのクリック確認

確認ポイント:

- 上部で「何のページか」がすぐ分かる
- ページが薄くても、候補者名・現職導線・公式情報が分かりやすい
- モバイルで横スクロールが出ない
- 日本語の改行が不自然でない
- 関連動向が候補予定者に混ざっていない
- 公式情報カードのタイトルとリンク先が一致している

## 13. 公式候補者一覧が出たあとの切り替え

1. `page_status.official_candidate_list_status` を `published` にする
2. `page_status.official_candidate_list` に公式 URL と確認日時を入れる
3. 見出しを `公式候補者` ベースに切り替える
4. `candidate_signals.status = official_candidate` へ必要分を更新する
5. `page_updates` に切り替え履歴を追加する
6. validate / generate / 実画面確認をやり直す

市長選では、候補者一覧が PDF だけで出ることもあるので、HTML ページの有無に引っ張られず、最初に出た公式一覧を採用する。

## 14. 迷ったときの優先順位

1. 誤認を防ぐ
2. 公式情報と報道ベースを混ぜない
3. 候補者を見分けやすくする
4. 薄い情報を薄いまま正直に見せる
5. 装飾より可読性を優先する

## 15. 機械化との付き合い方

- この project は立ち上げ期なので、最初から完全自動を目指さない
- ただし、毎回の手作業が将来の機械化につながるように記録を残す
- いまは `人が判断する -> md に理由を書く` を優先し、判断パターンがたまった段階で script / skill に落とす
- つまり、運用を無理にツールへ合わせるのではなく、運用から見えた反復パターンをあとで実装する

### 15.1 毎回残すもの

- どの公式ページを見て link / date を確定したか
- 県 source から市 source へ上書きしたかどうか
- どこで誤認しやすかったか
- そのイレギュラーが次回も起こりそうか

### 15.2 実装に回すタイミング

- 同じ判断手順が 2 回以上出たとき
- `ここまでは機械でよい / ここからは人が見る` の境界が言語化できたとき
- 過去メモを見返せば再現できる状態になったとき

この runbook の目的は、今すぐ全部を自動化することではなく、後で自動化できるように判断材料を取りこぼさないこと。
