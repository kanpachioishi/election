## 1. 目的

この文書は、選挙・地方政府サイトを継続的に充実させるための役割分担とレビューゲートを定義する。

この体制は、複数人チームだけでなく、1人と AI エージェントで運用する場合にも使う。  
重要なのは担当者の人数ではなく、`調査`、`データ化`、`実装`、`レビュー`、`公開判断` を混ぜないことである。

## 2. 基本方針

- 正確性を速度より優先する
- 公式ソースがない情報は主要表示に使わない
- 実装者とレビュー者の観点を分ける
- 作業単位ごとに、入力、出力、完了条件を先に決める
- 公開判断は気分ではなくチェック項目で行う
- `data/v1`、`scripts/current`、`site`、`docs` の責務を混ぜない
- 変更はローカルで検証し、まとまった単位でコミットする

## 3. チーム構成

### 3.1 マネージャー

役割:

- ロードマップと優先順位を管理する
- 作業を小さなチケットに分ける
- 担当レーンとレビュー担当を決める
- 作業間の依存関係を整理する
- 公開してよい状態かを判定する

成果物:

- 作業依頼票
- スプリント単位の優先リスト
- 公開判定
- 必要に応じた runbook 更新

### 3.2 調査・出典担当

役割:

- 公式ソースを探す
- URL、公開日、確認日、取得メモを残す
- 公式ソースと非公式情報を分ける
- 事実と推測を分ける

成果物:

- 公式ソース一覧
- 取得メモ
- 掲載可否の一次判断

止める条件:

- 公式ソースが見つからない
- 選挙名、地域、日付の対応が曖昧
- 報道、SNS、候補者本人情報しか確認できない

### 3.3 データ担当

役割:

- 調査結果を `data/v1` に正規化する
- ID、地域、選挙種別、日付、確認メタデータを揃える
- 生成データを直接編集しない
- validator のエラーを解消する

成果物:

- `data/v1` の更新
- 必要な生成データ
- 検証結果

止める条件:

- `Region` に安全に紐づけられない
- `source_url`、`confirmed_at`、`status` が欠ける
- `verified` として扱う根拠が弱い

### 3.4 実装担当

役割:

- `site`、`scripts`、`workers` の変更を実装する
- 既存の生成フローと表示ルールに合わせる
- 画面やデータの責務を増やしすぎない

成果物:

- ページ、UI、スクリプト、生成物
- 実行した検証コマンド
- 影響範囲メモ

止める条件:

- 仕様が未確定で表示が誤案内になり得る
- 生成物を正本として編集する必要が出た
- 検証コマンドが通らない

### 3.5 編集担当

役割:

- ページ本文、説明文、注記、見出しを整える
- 中立性と読みやすさを確認する
- 煽り、断定、根拠不明の評価を避ける
- SEO のために独自説明を厚くする

成果物:

- 公開可能な本文
- 出典付きの説明
- 注意書きと未確認表示

止める条件:

- 原文にない評価を断定している
- 候補者、政党、会派の支持不支持に見える
- 公式情報への導線がない

### 3.6 レビュー担当

役割:

- 事実、出典、データ、表示、操作、差分を確認する
- 実装者の意図ではなく、公開後の誤認リスクを見る
- 未確認情報が確定情報として表示されていないか確認する

成果物:

- レビュー記録
- 修正指摘
- 公開可否の推薦

原則:

- 実装担当とレビュー担当は分ける
- 1人運用で分けられない場合は、レビュー記録に `self_review: true` と明記し、チェック項目を省略しない

### 3.7 リリース担当

役割:

- 検証、生成、差分確認、コミット、デプロイを行う
- 公開前チェックが完了していることを確認する
- 公開後に主要ページを確認する

成果物:

- 公開判定
- コミット
- デプロイ後確認メモ

止める条件:

- validator が通らない
- 生成データが更新されていない
- レビュー未完了
- 公開してはいけない未確認情報が残っている

## 4. 作業単位

作業は原則として、次の単位に分ける。

| 作業種別 | 主担当 | 標準成果物 |
| --- | --- | --- |
| 選挙データ追加・修正 | 調査、データ | `data/v1/elections.json`、公式リンク |
| 公式リンク拡充 | 調査、データ | `data/v1/election_resource_links/*.json` |
| 個別選挙ページ追加 | 実装、編集 | `site/elections/*.html`、必要な assets |
| 台帳ページ追加・更新 | データ、実装 | canonical JSON、生成ページ |
| 市議会・行政コンテンツ | 調査、編集 | 草稿、出典表、公開ページ |
| スクリプト改善 | 実装 | `scripts/current`、検証結果 |
| 公開作業 | リリース | コミット、デプロイ確認 |

## 5. 標準フロー

1. マネージャーが作業依頼票を作る
2. 調査・出典担当が公式ソースを確認する
3. データ担当が canonical データへ正規化する
4. 実装担当がページやスクリプトへ反映する
5. 編集担当が文言と中立性を整える
6. レビュー担当がレビュー記録を作る
7. リリース担当が公開判定を行う

小さな修正では 1 人が複数役割を兼ねてよい。  
その場合も、作業の順番とチェック観点は省略しない。

## 6. ゲート

### Gate A: 調査ゲート

通過条件:

- 公式ソース URL がある
- 参照日または確認日がある
- 対象選挙、地域、日付の対応が説明できる
- 非公式情報を公式リンクとして扱っていない

### Gate B: データゲート

通過条件:

- `data/v1` の正本に入っている
- `source_url`、`source_type`、`confirmed_at`、`last_checked_at`、`status` がある
- `status = verified` の根拠がある
- `node scripts/current/validate-data-v1.mjs` が通る

### Gate C: 表示ゲート

通過条件:

- 表示上、未確認情報を確定情報に見せていない
- 出典、確認日、公式リンクが見える
- スマホで読める構成になっている
- 生成データを直接編集していない

### Gate D: 公開ゲート

通過条件:

- レビュー記録がある
- 必要な検証コマンドが通っている
- `git diff` で意図しない差分がない
- 公開後に確認する URL が決まっている

## 7. 標準検証コマンド

データを触った場合:

```bash
node scripts/current/validate-data-v1.mjs
node scripts/generate-site-data.mjs
node scripts/current/validate-data-v1.mjs
```

フロントの JavaScript を触った場合:

```bash
node --check site/assets/app.js
```

公開前:

```bash
git status --short
git diff --stat
```

必要に応じて、対象ページをローカルサーバーで確認する。

```bash
python3 -m http.server 4173 --directory site
```

## 8. 作業依頼票の保存場所

公開に直接関係しない作業中メモは、原則として `artifacts/private/batches/operations/` に置く。  
長期的に参照する運用ルール、計画、仕様は `docs/` に移す。

推奨ファイル名:

```text
artifacts/private/batches/operations/work-YYYYMMDD-{slug}.md
```

例:

```text
artifacts/private/batches/operations/work-20260428-expand-upcoming-mayor-links.md
```

## 9. 優先順位の付け方

優先順位は次の順で決める。

1. 誤案内や古い情報を防ぐ作業
2. upcoming 選挙の公式リンク拡充
3. 郵便番号検索と地域ページの完成
4. 個別選挙ページの拡大
5. 読み物・解説・議会コンテンツの拡充
6. 通知、PWA、外部連携などの習慣化機能

## 10. マネージャー判断が必要なケース

- 公式ソース同士で情報が食い違う
- 非公式情報をどこまで扱うか迷う
- 候補者、政党、会派への評価に見える表現がある
- 生成フローを変える必要がある
- 公開範囲を広げるが検証体制が追いつかない
- 1つの変更が `data`、`scripts`、`site`、`workers` をまたぐ

## 11. Done 条件

作業は、次を満たしたときに完了とする。

- 作業依頼票の完了条件を満たしている
- 必要な公式ソースが記録されている
- canonical データと生成物の責務が守られている
- 検証コマンドが通っている
- レビュー記録がある
- 公開する場合は公開判定がある
- 次に確認すべき未完了事項が残っている場合、明示されている
